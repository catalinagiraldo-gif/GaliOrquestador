## 1. Diagnóstico — Por qué no existe esta vista hoy

### 1.1 Ausencia total del enfoque predictivo

El botón "Señales" existe en el Hub pero no tiene destino. Lo que hay en Decisiones pendientes son **alertas puramente reactivas** (novedad de Coordinadora, cambio de transportadora). No hay ninguna pantalla que muestre lo que Gali ve *antes* de que el problema ocurra.

Cita directa de CorrecionesGali5.md:
> "La mitad del valor diferencial de Gali — el carácter predictivo — no se prototipa. No hay ningún ejemplo de lo que se definió como señal: 'tendencia de mercado que se abre en tu segmento', 'oportunidad de escalar antes de que llegue la competencia'."

### 1.2 Data del macromundo de Dropi invisible

El diferenciador más fuerte de Gali vs. cualquier IA genérica es que cruza datos globales de toda la red Dropi con el micromundo del usuario. Hoy ese cruce es invisible: las recomendaciones de ADA Spy aparecen sin decir de dónde vienen.

El patrón correcto (GaliAjustesJun3.md Bloque 2):
```
Macromundo: "Coordinadora 15% novedad en Bogotá esta semana"
Micromundo: "Este usuario tiene 23 pedidos activos hacia Bogotá"
Output:     "Cambié 23 pedidos a Servientrega. Ahorré ~4 novedades."
```

Sin el primer elemento visible, el usuario no entiende por qué Gali hace lo que hace.

### 1.3 Loop cerrado sin ruta propia

DesADIGali5.md §8: La promesa "Vista Señales diferenciada" está en estado **Parcial** — existen cards `gali-signal-card-v2` con tipos (crítica, oportunidad, completada) en el Hub, pero no hay ruta propia, y chat y señales compiten en el mismo layout.

El loop de acción (AlcancesIADropi.md VI) requiere una superficie dedicada:
```
Señal detectada
    → Acción tomada (con contexto de por qué)
    → Resultado medible (before / after)
    → Siguiente paso propuesto (crear skill, ajustar regla)
```

Sin ruta propia, este loop no puede ocurrir sin que el usuario abandone el Hub.

## 2. Distinción Señales vs. Alertas

Esta distinción es la columna vertebral del componente. Son dos conceptos fundamentalmente distintos y deben verse distintos:

| Dimensión | SEÑAL | ALERTA |
|---|---|---|
| **Naturaleza** | Predictiva — algo que *va a pasar* o *puede aprovecharse* | Reactiva — algo que *está pasando* y requiere acción |
| **Urgencia** | Ventana de 7-14 días | Requiere acción hoy o esta semana |
| **Origen** | Data global de la red Dropi cruzada con el micromundo del usuario | Estado actual de pedidos, campañas, conexiones |
| **Agente** | ADA Spy, Roax (cuando detectan tendencias antes de actuar) | Vigilante, Chatea Pro, Kronos (cuando detectan problemas activos) |
| **Tono** | Estratégico, orientado a oportunidad | Urgente, orientado a solución |
| **CTA** | "Aprovechar", "Explorar", "Aprobar escala" | "Resolver ahora", "Cambiar transportadora", "Conectar" |
| **Fondo visual** | `var(--surface-50)` — blanco muy sutil, tono neutro-cálido | `var(--orange-50)` o `var(--red-50)` según criticidad |

### 5.2 `senales-page.component.ts`

```typescript
import { Component, signal, computed } from '@angular/core';
import { CommonModule } from '@angular/common';
import { Router } from '@angular/router';
import { TooltipModule } from 'primeng/tooltip';
import { MOCK_SENALES, MOCK_ALERTAS } from '../../../../../mocks/gali-v5/senales.mock';

type FilterTab = 'todas' | 'senales' | 'alertas' | 'completadas';

@Component({
  selector: 'app-senales-page',
  standalone: true,
  imports: [CommonModule, TooltipModule],
  templateUrl: './senales-page.component.html',
  styleUrls: ['./senales-page.component.scss'],
})
export class SenalesPageComponent {
  readonly senales = MOCK_SENALES;
  readonly alertas = MOCK_ALERTAS;

  activeFilter = signal<FilterTab>('todas');

  senalesVisibles = computed(() =>
    this.activeFilter() === 'alertas' ? [] :
    this.activeFilter() === 'completadas'
      ? this.senales.filter(s => s.tipo === 'completed')
      : this.senales.filter(s => s.tipo !== 'completed')
  );

  alertasVisibles = computed(() =>
    this.activeFilter() === 'senales' ? [] :
    this.activeFilter() === 'completadas' ? [] :
    this.alertas
  );

  setFilter(f: FilterTab) { this.activeFilter.set(f); }

  // Fase 1: log o toast. Fase 2 (post Spec 3): abrir ConfirmActionModalComponent
  handleAlertaCta(alerta: any) {
    console.log('alerta CTA', alerta.id);
  }

  handleSenalCta(senal: any) {
    // TODO Sprint 2: conectar acción real
    console.log('señal CTA', senal.id);
  }
}
```

### 13.1 Modal de confirmación para alertas críticas

`handleAlertaCta` en fase 1 es un `console.log`. En fase 2 abre el mismo modal de 2 pasos definido en Spec #3.

#### Decisión: reusar vs. extraer

El `ConfirmActionModalComponent` de Spec #3 se implementó como modal inline dentro del Hub. Para reutilizarlo desde Señales sin duplicar código, **extraerlo a un módulo compartido**:

```
src/app/shared/components/confirm-action-modal/
├── confirm-action-modal.component.ts
├── confirm-action-modal.component.html
└── confirm-action-modal.component.scss
```

Si ya existe en `pages/gali-v5/hub/`, moverlo a `shared/`. Si no existe aún, crearlo directamente en `shared/` y referenciarlo desde ambas rutas (Hub y Señales).

#### Interfaz del modal (compatibilidad con Spec #3)

```typescript
export interface ConfirmActionConfig {
  // Step 1 — qué va a pasar
  titulo: string;                  // "Cambiar transportadora"
  descripcion: string;             // "Voy a cambiar 12 pedidos de Coordinadora a Servientrega"
  impacto: string;                 // "~3 novedades evitadas · ~$51k en riesgo"
  iconoAgente?: string;            // 'vigilante' | 'roax' | 'ada' | 'kronos'

  // Step 2 — cuánto afecta
  itemsAfectados?: number;         // 12 pedidos
  labelItems?: string;             // "pedidos"
  detailList?: string[];           // lista opcional de IDs o nombres afectados

  // CTAs
  ctaConfirmar: string;            // "Sí, cambiar 12 pedidos"
  ctaCancelar: string;             // "Cancelar"

  // Callbacks
  onConfirm: () => void;
  onCancel?: () => void;
}
```

#### Cambio en `SenalesPageComponent`

```typescript
// senales-page.component.ts
import { ConfirmActionConfig } from '../../../../../shared/components/confirm-action-modal/confirm-action-modal.types';

showConfirmModal = signal(false);
confirmConfig = signal<ConfirmActionConfig | null>(null);

handleAlertaCta(alerta: GaliAlerta) {
  const config: ConfirmActionConfig = {
    titulo: this.getModalTitulo(alerta),
    descripcion: alerta.descripcion,
    impacto: alerta.impacto,
    iconoAgente: alerta.agente,
    itemsAfectados: alerta.pedidosAfectados ?? undefined,
    labelItems: alerta.pedidosAfectados ? 'pedidos' : undefined,
    ctaConfirmar: alerta.ctaPrincipal,
    ctaCancelar: 'Cancelar',
    onConfirm: () => {
      this.showConfirmModal.set(false);
      // Fase 3: llamada real al servicio
      console.log('Acción confirmada para alerta', alerta.id);
    },
    onCancel: () => this.showConfirmModal.set(false),
  };
  this.confirmConfig.set(config);
  this.showConfirmModal.set(true);
}

private getModalTitulo(alerta: GaliAlerta): string {
  const titulos: Record<string, string> = {
    'alt-001': 'Cambiar transportadora',
    'alt-002': 'Reconectar integración',
    'alt-003': 'Activar creative alternativo',
  };
  return titulos[alerta.id] ?? 'Confirmar acción';
}
```

```html
<!-- senales-page.component.html — al final del template -->
<app-confirm-action-modal
  *ngIf="showConfirmModal()"
  [config]="confirmConfig()!"
  (closed)="showConfirmModal.set(false)">
</app-confirm-action-modal>
```

#### Regla de negocio: qué alertas abren modal

Solo abren el modal de confirmación las alertas con `tipo === 'critical'` que involucran **acciones irreversibles o de alto impacto** (cambiar pedidos, reconectar integración). Las alertas `warning` e `info` navegan directo (sin modal) o ejecutan acción lightweight.

```typescript
handleAlertaCta(alerta: GaliAlerta) {
  if (alerta.tipo === 'critical') {
    this.abrirModalConfirmacion(alerta);
  } else {
    // warning / info: acción directa sin confirmación
    console.log('Acción directa alerta', alerta.id);
  }
}
```

### 13.3 Botón "Operar" en `gali-workspace-mode-bar` → `/senales` con filtro alertas

El mode-bar tiene botones de modo (`Analizar`, `Operar`, `Planificar`). "Operar" está vinculado al ámbito operativo — alertas activas que requieren acción.

#### Estrategia: query param `?filtro=alertas`

Al navegar desde el mode-bar se pasa un query param para pre-seleccionar el filtro:

```typescript
// En el componente que contiene el mode-bar (home.component.ts o workspace-mode-bar.component.ts)
onModeOperar() {
  this.router.navigate(['/gali-v5/senales'], {
    queryParams: { filtro: 'alertas' }
  });
}
```

#### `SenalesPageComponent` lee el query param al inicializar

```typescript
// senales-page.component.ts
import { ActivatedRoute } from '@angular/router';

constructor(private route: ActivatedRoute, private router: Router) {
  // Leer filtro inicial desde query param
  const filtroParam = this.route.snapshot.queryParamMap.get('filtro') as FilterTab | null;
  if (filtroParam && ['todas','senales','alertas','completadas'].includes(filtroParam)) {
    this.activeFilter.set(filtroParam);
  }
}
```

**Por qué `snapshot` y no `queryParams` observable**: la ruta no se recarga con el mismo componente en este contexto, `snapshot` es suficiente y evita una suscripción innecesaria.

#### Template del mode-bar (si tiene `data-proto-skip`)

```html
<!-- workspace-mode-bar o home — botón Operar -->
<!-- Quitar data-proto-skip -->
<button
  class="mode-btn"
  [class.mode-btn--active]="activeMode === 'operar'"
  (click)="onModeOperar()">
  <img src="assets/icons/sidebar/operar.svg" alt="" width="20" height="20" />
  Operar
</button>
```

#### Comportamiento al volver al Hub desde Señales

Cuando el usuario navega de vuelta al Hub (botón back o sidebar), el filtro no persiste — es solo entrada. No hace falta limpieza adicional porque el componente se recrea con `snapshot` en cada navegación.
