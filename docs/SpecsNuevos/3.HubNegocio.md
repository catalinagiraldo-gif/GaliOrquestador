## 1. Diagnóstico — Por qué el Hub actual no funciona

### 1.1 Sobrecarga cognitiva (Heurística #8)

El Hub acumula **7 bloques de contenido** compitiendo simultáneamente sin jerarquía visual clara:
barra de intent global · tabs de dashboard · banner de objetivo · banner de alerta de agente · ciclo de negocio (6 tarjetas) · Decisiones pendientes (cards con acciones) · chat de Gali con sugerencias · Akademy recomendado.

El usuario no sabe qué atender primero. La heurística de minimalismo exige reducir la densidad sin perder funcionalidad.

### 1.2 Acciones masivas sin confirmación (Heurística #3)

Los botones "Cambiar todos → Servientrega" y "Cambiar solo los de hoy" en Decisiones pendientes ejecutan acciones sobre pedidos activos sin ningún paso de confirmación. Redirigir 12+ pedidos de transportadora en tiempo real tiene consecuencias financieras directas. Un clic accidental es crítico e irreversible.

### 1.3 Ciclo de negocio — layout roto (Problema #2)

Las 6 tarjetas del ciclo de negocio (Producto → Marketing → Pedidos → Logística → Finanzas → Reportes) no fluyen linealmente:
- Logística aparece desfasada hacia abajo y a la derecha
- No hay flechas conectoras entre todas las tarjetas
- Hay dos columnas completamente vacías sin estado explícito — el usuario no sabe si es un error o contenido pendiente

### 1.4 Dos puntos de entrada al chat de Gali (Problema #5 + "Reformular Hub")

La barra de intent superior (`"¿Qué quieres hacer?"`) y el panel lateral `"Hablar con Gali"` hacen exactamente lo mismo. La redundancia confunde sobre cuál usar y ocupa espacio visual valioso que podría usarse para contenido de mayor urgencia.

### 1.5 ROAS incorrecto en tarjeta de Marketing

La tarjeta de Marketing del Hub muestra `2.9x` como ROAS — ese es el valor **declarado por Meta Ads**, no el ROAS real de Dropi. El valor real según el mock maestro (`projects.json`) es `1.93x`. Mostrar el dato de Meta como si fuera el real distorsiona la lectura del negocio.

### 1.6 "Ciclo de negocio" como micro-label invisible (Problema #12)

La etiqueta `"↻ Ciclo de negocio"` aparece como texto muy pequeño junto al header. No guía al usuario sobre cómo leer el flujo central de la pantalla.

### 1.7 Sin ROI acumulado de Gali (Bloque 2 #1 + #8)

No existe ningún indicador que muestre qué hizo Gali automáticamente vs. qué requiere decisión del usuario. El Hub se percibe como un dashboard tradicional de KPIs, no como un orquestador IA que ejecuta acciones por el usuario. Sin ese número de valor percibido, el dropshipper no puede justificar emocionalmente que Gali es indispensable.

## 2. GenUI Progresiva — El Hub en 3 estados (Bloque 6 de GaliAjustesJun3)

El Hub no debe tener una densidad fija. Su layout se revela progresivamente según el nivel del usuario:

| Estado | Perfil | Densidad del Hub |
|---|---|---|
| **Estado Cero** | Usuario nuevo, sin proyectos activos | 3 módulos visibles máximo · Onboarding conversacional · Sin ciclo de negocio (vacío) |
| **Estado Activo** | Usuario con 1-3 proyectos en curso | Las 3 zonas de esta spec · Alertas consolidadas · Ciclo de negocio con datos reales |
| **Estado Experto** | Operador con 3+ proyectos, múltiples agentes | Todas las zonas + toggle "Vista experto" que expande sub-métricas · Gali propone layout óptimo |

**Esta spec define el layout para Estado Activo** — que es el estado del usuario demostrado en el prototipo (47 pedidos/sem, 2+ proyectos).

El botón `"← Vista simple"` existente pasa a ser el toggle entre Estado Activo y Estado Cero. `"↺ Reconfigurar Gali"` abre un modal de configuración del workspace — no resetea todo el sistema.

## 3. Nueva Estructura del Hub — Tres Zonas

```
┌──────────────────────────────────────────────────────────────────────┐
│  BARRA DE INTENT GLOBAL (se mantiene, única entrada al chat de Gali) │
├──────────────────────────────────────────────────────────────────────┤
│  ZONA 1 — DECISIONES PENDIENTES + SEÑALES DE AGENTES                │
│  Mayor peso visual · Acción requerida del usuario                    │
├──────────────────────────────────────────────────────────────────────┤
│  ZONA 2 — CICLO DE NEGOCIO (título visible)                          │
│  Producto → Marketing → Pedidos → Logística → Finanzas → Reportes   │
│  Flechas conectoras entre todas · Estado explícito en celdas vacías  │
├──────────────────────────────────────────────────────────────────────┤
│  ZONA 3 — GALI ESTA SEMANA + AKADEMY (pill)                          │
│  Colapsada por defecto · Chat como drawer lateral desde header       │
└──────────────────────────────────────────────────────────────────────┘
```

### Separación visual entre zonas

Cada zona debe ser visualmente distinguible sin necesidad de leer los títulos:
- Zona 1: borde izquierdo naranja `2px solid var(--primary-500)` + fondo `var(--surface-50)` (off-white muy sutil, no blanco puro)
- Zona 2: fondo blanco, título de sección prominente (ver §5)
- Zona 3: fondo `var(--surface-100)`, menor altura visual, ligeramente más oscura

## 4. Zona 1 — Decisiones Pendientes + Señales de Agentes

### 4.1 Layout de Zona 1

```
┌────────────────────────────────────────────────────────────┐
│  🔴 Decisiones pendientes (2)          [Agentes activos]   │
│  ──────────────────────────────────────────────────────    │
│  [card 1]  Coordinadora Bogotá: 15% novedad hoy            │
│            Vigilante recomienda cambio de transportadora   │
│            [Cambiar todos →]  [Solo los de hoy]            │
```

- El título "Decisiones pendientes (N)" tiene un badge con el conteo de decisiones activas
- Los agentes activos se muestran como pills compactos a la derecha: `● Vigilante activo` · `● Roax activo`
- Las cards de decisión tienen jerarquía visual por urgencia: borde izquierdo `var(--red-400)` para crítico, `var(--amber-400)` para recomendación

### 4.2 Modal de confirmación de 2 pasos (Heurística #3)

**Todo botón de acción masiva en Zona 1 abre un modal antes de ejecutar.**

El modal tiene 2 pasos:

**Paso 1 — Contexto de la acción**:
```
┌─────────────────────────────────────────────────────┐
│  Cambiar transportadora en pedidos activos          │
│  ─────────────────────────────────────────────────  │
│  Vigilante detectó: Coordinadora Bogotá 15% novedad │
│                                                     │
│  Esta acción afecta:                                │
│  • 12 pedidos activos con destino Bogotá            │
│  • Transportadora destino: Servientrega             │
│  • Impacto estimado: evitar ~3 novedades ($51k)     │
│  • Tiempo de gestión: ~2 horas adicionales          │
│                                                     │
│  [Cancelar]                    [Revisar y confirmar →]│
└─────────────────────────────────────────────────────┘
```

**Paso 2 — Confirmación final**:
```
┌─────────────────────────────────────────────────────┐
│  Confirmar cambio                                   │
│  ─────────────────────────────────────────────────  │
│  ✓ 12 pedidos · Bogotá · → Servientrega             │
│                                                     │
│  Esta acción no se puede deshacer.                  │
│  Vigilante notificará cuando esté completo.         │
│                                                     │
│  [← Volver]              [Confirmar cambio ahora]   │
└─────────────────────────────────────────────────────┘
```

**Specs del modal**:
- Overlay con `backdrop: blur(4px) rgba(0,0,0,0.3)`
- Ancho máx: 480px, centrado vertical y horizontalmente
- Botón de confirmación: color de fondo `var(--primary-500)` (#f49a3d), texto blanco
- Botón destructivo (si aplica): borde `var(--red-500)`, texto rojo, fondo blanco
- El modal sigue las reglas de `dropi-modal.json` del DS Registry

**Acciones que SIEMPRE requieren este modal**:
- Cualquier cambio que afecte más de 1 pedido
- Escalar o pausar pauta publicitaria
- Cambiar transportadora en lote

Justo debajo de las cards de Decisiones, antes de Zona 2, un bloque compacto de 3 líneas que muestra el ROI acumulado de Gali:

```

│  ✦ Gali esta semana                                    │

│  ⚡ 23 acciones ejecutadas automáticamente             │
```

## 5. Zona 2 — Ciclo de Negocio (rediseño)

### 5.1 "Ciclo de negocio" como título de sección visible

El micro-label `"↻ Ciclo de negocio"` se convierte en un **título H2 de sección** prominente:

```html
<div class="section-header">
  <h2 class="section-title">Ciclo de negocio</h2>
  <span class="section-subtitle">
    Flujo de tu operación esta semana
  </span>
  <button class="section-info-btn" pTooltip="Cómo leer este flujo" tooltipPosition="right">
    <i class="pi pi-info-circle"></i>
  </button>
</div>
```

**Specs tipográficas** (DS tokens):
- `h2.section-title`: Inter SemiBold 16px, `var(--gray-900)`
- `span.section-subtitle`: Inter Regular 12px, `var(--gray-500)`
- Separación del header al grid de tarjetas: `$size-4` (16px)

### 5.2 Layout horizontal con flechas conectoras

Las 6 tarjetas deben organizarse en **una fila horizontal** con flechas entre todas:

```
[Producto] → [Marketing] → [Pedidos] → [Logística] → [Finanzas] → [Reportes]
```

- Si el viewport no da el ancho, el contenedor tiene `overflow-x: auto` con `scroll-snap-type: x mandatory`
- Las flechas conectoras son SVG inline con `var(--gray-300)`, 12px de alto, posicionadas en el gap entre tarjetas

**Solución para las dos columnas vacías**:

Las celdas vacías (actualmente sin estado) deben mostrar un estado explícito:

```html
<!-- tarjeta en estado vacío -->
<div class="cycle-card cycle-card--empty">
  <div class="empty-icon">
    <img src="assets/icons/sidebar/gali-v5/boxes.svg" width="20" height="20" />
  </div>
  <p class="empty-label">Sin datos aún</p>
  <p class="empty-cta">Conecta tu catálogo</p>
</div>
```

- Fondo: `var(--surface-100)` con borde dashed `var(--gray-200)`
- Texto vacío: Inter Regular 12px, `var(--gray-400)`
- Sin estado de error — es un estado neutral de "aún no conectado"

### 5.3 ROAS correcto en tarjeta Marketing

**Antes** (incorrecto): `ROAS 2.9x`

**Después** (correcto — del mock):
```html
<div class="cycle-metric">
  <span class="metric-value">1.93x</span>
  <span class="metric-label">ROAS real</span>
```

**Valores correctos por tarjeta del ciclo** (del mock `collar-gps-2026` como proyecto principal):

## 6. Zona 3 — Gali Esta Semana + Akademy (colapsada)

### 6.1 Zona 3 colapsada por defecto

La Zona 3 arranca contraída y muestra solo un handle visual:

```
──────────────────────────────────────────────────────────
  ▸ Academy · Personalizado para ti · [1 curso nuevo]
──────────────────────────────────────────────────────────
```

Al expandir, muestra:
- Un pill de "Akademy" con el curso recomendado por Gali (enlace a `/gali-v5/academy`)
- El bloque "Gali esta semana" se mueve aquí **solo si la Zona 1 ya está vacía** (sin decisiones pendientes) — de lo contrario permanece en Zona 1

**Transición de colapso**: `max-height` de `40px` → `180px`, 200ms ease-out.

### 6.2 Chat de Gali — eliminar redundancia (Problema #5)

**Problema actual**: La barra de intent superior y el panel `"Hablar con Gali"` son el mismo punto de entrada.

**Solución**:
- La **barra de intent superior** (`"¿Qué quieres hacer?"`) se mantiene como el **único** punto de entrada al chat de Gali
- El panel lateral `"Hablar con Gali"` se convierte en un **drawer** que se abre desde el header (botón ícono de Gali, ya existente)
- El drawer se desliza desde la derecha: `transform: translateX(100%)` → `translateX(0)`, 250ms ease-in-out
- El drawer tiene ancho fijo 360px y `position: fixed` con `z-index: 1000`
- La barra de intent en el Hub puede tener un CTA secundario `"Abrir chat completo →"` que abre el drawer

**Archivos afectados**:
- `home.component.html`: eliminar el bloque del panel `"Hablar con Gali"` del layout del Hub
- El componente del drawer (si ya existe como `gali-right-panel`) se activa solo desde el header, no desde el Hub
