## 0. Restricción de producto (léase primero)

Antes de cualquier otra cosa, la restricción que originó este documento: aunque la UI vaya a mostrar varios
motores de IA como "conectables" (Claude, Gemini, GPT...), **no hay conexión real a ningún proveedor de IA
externo en el prototipo**. Cualquier respuesta compleja real que el prototipo llegue a resolver de verdad se
resuelve con **Claude Haiku** por debajo — igual que hoy Meta Ads, Siigo o WhatsApp Business en
`gali6-conexiones-casa.component.ts` son fachada/mock, no integraciones reales. Todo lo que sigue en este
documento diseña esa fachada con la misma honestidad de producto: nunca prometer una capacidad de routing
entre modelos que el prototipo no puede sostener.

---

## 1. Visión

Gali sigue siendo el único "cerebro" que razona — hoy y en cualquier fase futura de este documento, ese
cerebro es Claude Haiku (real, si algún día se conecta algo de verdad al prototipo). Lo que el usuario
conecta en esta spec **no son cerebros alternativos**, son **fuentes de contexto y ejecución adicionales**,
exactamente como ya conecta Meta Ads o Shopify hoy en Conexiones.

Esto resuelve dos cosas a la vez:

1. **Evita prometer algo que el prototipo no puede sostener** — no hay "elige tu LLM favorito" real, porque
   eso implicaría credenciales, facturación y latencia por proveedor que este prototipo no maneja.
2. **Reutiliza un patrón que el usuario ya entiende.** Diana lo resumió en la transcripción del 2 de junio:
   *"herramientas que se conectan a los agentes... el conocimiento se puede llevar a otro nivel"* — un motor
   de IA es, para efectos de la UI, una tarjeta más en la misma pantalla donde ya viven Siigo o WhatsApp
   Business, no una sección nueva con reglas propias.

## 2. Modelo de conexión — nueva sub-vista "Motor de Gali" dentro de Conexiones

### 2.1 Por qué una sub-vista, no una tarjeta más

La versión anterior de este documento proponía extender `Mcp.tipo` con `'ia'` y tratar cada motor como una
tarjeta más del grid plano. Un mockup real de Catalina mostró algo distinto: un panel de configuración técnica
del motor de Gali con datos que **no caben en el shape de `Mcp`** (`glyph/dato/contexto/campanasActivas`,
pensado para integraciones de negocio tipo Meta Ads o Siigo) — barra de uso de tokens, selector de modelo
rápido/pro, y servidores MCP con conteo de tools y toggle on/off. Forzar esos datos en una tarjeta habría
significado inventar campos opcionales sueltos sobre una interfaz que no fue diseñada para eso.

La corrección: **una sub-vista propia**, no un componente nuevo desde cero. Dentro de la tab "Conexiones" de
Centro de Gali (`gali6-conexiones-casa.component.ts`), un segmented-control de 2 opciones sobre el grid
existente:

- **"Integraciones de negocio"** (default) — el grid plano actual (`mcps[]`), sin ningún cambio.
- **"Motor de Gali"** — panel nuevo con 3 tabs internos: **LLM / Tools / MCPs**.

El flujo de conexión ya existente (`abrirPanel`, `apiKeyInput`, `conectarMcp`, `getPasosConexion`, simulación
de 1800ms) se **reutiliza tal cual** dentro de la tab LLM — no se inventa un mecanismo de conexión nuevo, solo
una superficie distinta para mostrarlo.

### 2.2 Tab LLM

**Proveedor de IA** — dos cards, igual que el mockup:

| Card | Estado | Texto |
|---|---|---|
| IA de Dropi | seleccionado por defecto | *"Pago por consumo · sin configuración"* |
| Mi propia IA | seleccionable | *"Conecta tu API key · usas tu cuenta"* |

Clic en "Mi propia IA" expande el selector de proveedor específico — aquí es donde se reusa la tabla de la
versión anterior de este documento, sin perderla, solo un nivel más adentro:

| `nombre` | `estado` inicial | `dato` |
|---|---|---|
| Claude | `conectado` | *"Motor por defecto de Gali — ya resuelve todas tus consultas."* |
| Gemini | `pendiente` | *"Conéctalo para comparar respuestas o usarlo en skills específicas."* |
| GPT | `pendiente` | *"Conéctalo para comparar respuestas o usarlo en skills específicas."* |
| Llama / Mistral | `proximamente` | *"Próximamente — modelos open-source para tareas de bajo costo."* |

Clic en cualquiera de estas → `abrirPanel(mcp)` (mecanismo real ya existente) → pega API key ficticia →
`conectarMcp()` (ya existente, simula 1800ms) → estado pasa a `conectado`. **Nota de producto honesta y
explícita en el panel** (la restricción de §0, repetida en el lugar donde el usuario puede malinterpretar lo
que está pasando): *"Esta conexión no envía datos reales a Gemini. Todas las respuestas hoy las resuelve
Claude Haiku."* — misma fachada que ya aplica a Meta Ads, Siigo, etc.

**Uso de tokens** — barra de progreso (ej. *"12.340 / 100.000 tokens · 12% · Uso de este mes. Se reinicia el
día 1."*). Mismo mock que describe §3 — vive aquí, no en una tarjeta.

**Modelo** — dos cards, solo relevantes cuando el proveedor activo es "IA de Dropi": *"Dropi IA · Rápido"*
(*"Respuestas rápidas, menor consumo"*) y *"Dropi IA · Pro"* (*"Mayor calidad y razonamiento"*). Nota de
producto — misma de §0: elegir "Pro" no cambia el motor real (sigue siendo Haiku), es una superficie de
producto para cuando exista diferenciación real de modelo.

### 2.3 Tab Tools

Lista simple de capacidades habilitadas para el asistente (mock nuevo, sin backend real detrás — mismo
criterio de honestidad de §0): ej. *"Buscar en catálogo"*, *"Consultar pedidos"*, *"Generar reporte"*, cada una
con un toggle on/off. Esta tab documenta qué puede "tocar" Gali cuando responde, no qué modelo usa — es el
complemento natural de la tab LLM, no un tercer proveedor de IA.

### 2.4 Tab MCPs

Servidores MCP conectados — el asistente expone las tools de cada uno cuando está activo:

```ts
interface McpServer {
  nombre: string;
  url: string;           // ej. "mcp.dropi.co" o "localhost:3000"
  toolsCount: number;
  estado: 'conectado' | 'desconectado';
  builtin?: boolean;     // true solo para "Dropi MCP" — no se puede desconectar
}
```

"Dropi MCP" aparece siempre, `builtin: true`, `toolsCount: 8`, toggle bloqueado en `conectado`. El usuario
puede agregar servidores propios vía **"+ Agregar servidor MCP"** (nombre + URL, mock — `toolsCount: 0` hasta
que "conecte", mismo patrón de simulación que `conectarMcp()`), cada uno con su propio toggle on/off.

## 3. Modelo de tokens

### 3.1 El precedente ya existe — no se inventa un mecanismo nuevo

`product-scout` en `mocks/gali-v6/agentes-especializados.ts` ya modela consumo de tokens a nivel de agente:

```ts
{
  id: 'product-scout',
  tier: 'tokens',
  tokensUso: 50,
  advertenciaIA:
    'Las recomendaciones de este agente se basan en patrones de tu historial y tendencias generales. ' +
    'No garantizan el éxito del producto — evalúa siempre con tu propio criterio.',
  // ...
}
```

Este documento **generaliza ese mismo patrón** a nivel de conexión en vez de a nivel de agente individual: en
vez de "este agente cuesta 50 tokens por uso", la **tab LLM** del panel "Motor de Gali" (§2.2) muestra:

- **Consumo del período** (mock, ej. *"12.340 / 100.000 tokens · 12%"*, tal como el mockup).
- **Límite mensual configurable por el usuario** — un input numérico simple en el panel, sin backend real
  detrás (se guarda en `localStorage`, mismo patrón que `gali-6-objetivo-meta` en `gali-6-shell.component.ts`).
- **Alerta de saldo bajo** — mismo componente visual `dropi-tag`/badge que ya usa `estado: 'urgente'` en las
  tarjetas MCP existentes (ej. Siigo), disparada cuando el consumo del período supera ~80% del límite.

### 3.2 Regla de calibración: nunca inventar un número de confianza

Ningún dato de esta sección puede presentarse como si viniera de telemetría real. La cifra de "tokens usados"
es un mock coherente (incrementa de forma determinística con las acciones que el usuario ya toma en el
prototipo — mismo criterio que ya aplica `Gali6LiveMutationsService.version` para invalidar mocks), nunca un
número aleatorio en cada render. Esto extiende el mismo criterio que ya usan los agentes especializados vía
`advertenciaIA`: cuando Gali muestra una cifra que suena a medición real, el usuario tiene que poder confiar
en que es consistente entre una visita y la siguiente.

### 3.3 Gasto diferenciado por tipo de sección — la regla que ancla todo esto

`docs/SpecsNuevos/Gali6.md` §"Modelo de intervención de Gali por sección" ya define, para navegación, qué tan
proactiva es Gali según el tipo de pantalla. Ese mismo nivel de agencia decide **cuándo un motor de IA puede
gastar tokens sin preguntar primero**:

| Tipo de sección | Comportamiento de gasto de tokens | Ejemplo |
|---|---|---|
| Operativa, agencia alta (*"autónoma con reporte"*) | Gasta automático, sin pedir; reporta cuánto gastó junto al resultado visible | *"Usé 40 tokens de tu conexión Claude para esto. Te quedan 2.400 este mes."* |
| Configuración / stub (*"silenciosa"* / *"recuperación explicada"*) | Nunca gasta sin pedir explícitamente, aunque una respuesta con IA "enriquecería" el resultado | *"¿Quieres que use tu conexión de Gemini para revisar esto a fondo? Consumiría ~30 tokens."* [Sí, usar tokens] [No, gracias] |

Ambos casos reutilizan el patrón *preview-then-confirm* que ya implementa `Gali6ChatService.handleAction()`
para `pausar-campana::`, `corregir-direccion::`, etc. — el gasto de tokens es un `actionId` más, con la forma
`usar-tokens::<mcpId>::<cantidad>`, no un mecanismo de confirmación nuevo.

## 4. Selector de motor por tarea (model-routing UI)

`gali6-agentes.component.ts` ya tiene un slider de autonomía por agente (`autonomiaPct`, con
`getAutonomiaLabel()` mapeando 0–20 *"Solo notifica"*, 21–45 *"Pide aprobación"*, 46–70 *"Opera con límites"*,
71–100 *"Modo automático"*). Junto a ese slider, en el editor de reglas/skills, aparece un control nuevo:

> **"Motor para esta skill: `[Claude ▾]`"**

Con las mismas opciones que el selector de proveedor de la tab LLM (§2.2: Claude, Gemini, GPT, Llama/Mistral).
Nota de producto — la misma de §0,
repetida aquí porque es el punto donde más fácil es sobre-prometer: **la resolución real interna siempre es
Haiku.** Este selector es una superficie de producto para cuando exista routing real entre motores — hoy no
cambia ningún comportamiento del prototipo, es una configuración que se guarda pero no se lee en runtime.

## 5. Fases futuras

### 5.1 Fase 2 — El mapa de conexiones tipo grafo

En la transcripción del equipo (2 de junio, líneas 867–909), Catalina describió un "mapa de conexiones"
estilo Obsidian: nodos de qué IA/herramienta afecta qué parte de la operación. Ese prototipo **ya existe
parcialmente**: `ConexionesPageComponent` en `gali-v5/pages/conexiones/conexiones-page.component.ts` tiene
`agentNodes`/`platformNodes` tipados como `GraphNode`, con estado `connected` y flag `urgente` — pero no
incluye proveedores de IA todavía.

Fase 2 de esta spec: fusionar esa vista de grafo dentro de `gali6-conexiones-casa`, agregando los nodos de
motor de IA (§2) como un tercer tipo de nodo junto a `agentNodes` y `platformNodes`. **No se diseña en detalle
aquí** — queda explícitamente diferida, tal como ya lo declara `18.FlujoUsuarioGali6.md` §PARTE 3.

### 5.2 Fase 3 — Routing real (fuera de alcance de cualquier fase de prototipo)

Si alguna vez se conecta un proveedor de IA de verdad, el selector de §4 empezaría a tener efecto real, y el
modelo de tokens de §3 dejaría de ser mock. Esto requeriría credenciales por proveedor, facturación y
manejo de rate limits — explícitamente fuera del alcance de este documento y de cualquier fase de prototipo
actual.

## 6. Límites inamovibles (extiende `AlcancesIADropi.md` §VIII)

La lista de límites inamovibles de `docs/Conocimientos/AlcancesIADropi.md` §VIII se extiende con dos nuevos,
específicos de Conexiones IA:

- **Gali nunca envía datos del negocio del usuario a un proveedor de IA externo sin una conexión explícita y
  aprobada por el usuario.** Ningún proveedor de la tab LLM (§2.2) pasa a `estado: 'conectado'` sin que el
  usuario complete el flujo de `conectarMcp()` — no hay auto-conexión.
- **Gali nunca cambia el motor que resuelve una tarea sin que el usuario lo haya configurado así.** El
  selector de §4 es la única superficie que decide qué motor "aparenta" resolver una skill; Gali no lo cambia
  de forma autónoma ni con el pretexto de optimizar costo o latencia.

Ambos límites son una extensión directa de los ya existentes: *"Gali nunca comparte datos del negocio del
dropshipper con otros usuarios sin consentimiento explícito"* y *"Gali prefiere preguntar a adivinar cuando el
riesgo de error es alto"* — aplicados al nuevo recurso "conexión de IA" en vez de a datos de negocio en
general.

## 7. Fuera de alcance de este documento

- **Código.** Nada de §2–§4 está implementado — es puramente el documento de spec, como confirmó Catalina en
  `PlanChat.md` §Parte A.
- **El grafo tipo Obsidian** de §5.1 — diseño de fase futura, no de esta spec.
- **Routing real entre modelos** (§5.2) — depende de decisiones de infraestructura y costo que no le
  corresponden a un documento de producto.
- **Microcopy exacto de pantalla.** Si al implementar esto hace falta el texto final de la UI, hay que leer
  `gali6-conexiones-casa.component.html` real antes de escribirlo — no inventarlo a partir de este documento
  solo (mismo criterio que ya aplica `18.FlujoUsuarioGali6.md` §PARTE 3).

## 8. Verificación de completitud

1. ¿La restricción de "no hay conexión real, todo lo resuelve Haiku" aparece en cada punto donde el usuario
   podría malinterpretar capacidad? — Sí: §0, §2.2 (nota de producto del panel), §4.
2. ¿El modelo de tokens cita un precedente real del código en vez de inventar un mecanismo nuevo? — Sí: §3.1
   cita `product-scout` (`tier: 'tokens'`, `tokensUso: 50`, `advertenciaIA`) textual.
3. ¿El gasto de tokens está anclado a una regla ya establecida (el modelo de intervención por sección), no a
   una regla nueva e independiente? — Sí: §3.3 remite a `Gali6.md` §Modelo de intervención de Gali por sección.
4. ¿Cada mecanismo propuesto reutiliza UI/código ya existente en vez de proponer un componente nuevo? — Sí:
   §2.1/§2.2 (`abrirPanel`/`apiKeyInput`/`conectarMcp` de `gali6-conexiones-casa`), §3.3 (`handleAction`
   preview-then-confirm), §4 (slider de `gali6-agentes.component.ts`).
5. ¿Los límites inamovibles nuevos están anclados a los ya existentes en `AlcancesIADropi.md` §VIII, no
   inventados sueltos? — Sí, §6.
6. ¿El diseño de §2 refleja el mockup real de Catalina (LLM / Tools / MCPs) en vez de forzar los datos nuevos
   en la interfaz `Mcp` existente? — Sí, corregido en el changelog del encabezado y en §2.1-§2.4.
