## 1. Diagnóstico — Lo que está mal hoy

### 1.1 Estado Cero inexistente (CorrecionesGali5.md Bloque 3)

El prototipo asume desde el primer frame que el usuario tiene operación activa (47 pedidos/sem, ROAS 2.9x, productos en escala). No hay ninguna pantalla, flujo ni estado para alguien que acaba de entrar a Dropi, no tiene pedidos y necesita que Gali le enseñe por dónde empezar. La visión habla de "hiperpersonalización desde el inicio" pero el prototipo solo la resuelve para usuarios maduros.

### 1.2 Onboarding no diferencia nuevo vs. veterano

El `GaliGoalOnboardingComponent` actual siempre muestra los 5 pasos completos, incluso si el usuario lleva un año en Dropi con 100 pedidos/semana. El principio de GaliAjustesJun3 Bloque 1 es claro:

> "Gali no te pregunta lo que Dropi ya sabe."

Si el usuario lleva 6+ meses y tiene historial, Gali ya sabe quién es. Preguntarle de nuevo es una fricción innecesaria que rompe la confianza.

### 1.3 Toggle de complejidad enterrado (Problema #11 CorrecionesGali5.md)

`GaliWorkspaceService` ya tiene `complexityLevel` con `setComplexityLevel()` y persiste en localStorage. El botón "← Vista simple / ↺ Reconfigurar Gali" existe **al fondo del scroll del Hub**, donde nadie llega. No hay UI prominente para cambiar de modo. El toggle debe estar visible en la zona superior del Hub, no como footer.

### 1.4 Tabs de proyecto con placeholder inutilizable (Problema #5 · DesADIGali5.md §6.3)

Los tabs Producto, Estrategia, Campañas y Pedidos en `proyecto-detalle-page.component.html` L.268-301 muestran:
- Un ícono genérico
- Un texto que dice "Esta sección conecta con [Catálogo/Roax/Mis Pedidos] con vista filtrada para este proyecto."
- Un botón "Ir al módulo →"

Desde la perspectiva del dropshipper, entrar a "Pedidos" dentro de su proyecto y encontrar una pantalla vacía que lo saca del proyecto es un quiebre de flujo grave. Los datos existen en `projects.json` — solo falta mostrarlos filtrados por `proyectoId`.

## 2. Principio de diseño: UX Relacional y Progresiva

Esta spec aplica el principio **agentic-ux-design-relationship-centric-interfaces**:

- Gali recuerda al usuario — no le repregunta lo que ya sabe
- La relación tiene historia: sesiones previas, objetivos declarados, comportamiento observado
- La interfaz se adapta a la **etapa de la relación**, no a una plantilla fija
- El onboarding no es un formulario — es una conversación que parte del contexto existente

**Los 3 estados de relación** (GaliAjustesJun3 Bloque 6):

| Estado | Señal de detección | UI del Hub |
|---|---|---|
| **Cero** | `sessionCount === 0` + `pedidosPerWeek === 0` | 3 preguntas conversacionales → workspace con 2-3 módulos |
| **Activo** | `sessionCount > 0` + `pedidosPerWeek > 0` | Tres zonas completas (Spec #3) |
| **Experto** | `sessionCount > 10` + `mesesOperando === '1y+'` + `pedidosPerWeek > 50` | Zona experto + toggle visible + Centro de Gali desbloqueado |

## 3. Onboarding — Dos caminos según el perfil

### 3.1 Lógica de detección de perfil

**Agregar a `GaliGoalOnboardingComponent`** o **a la función `shouldShowOnboarding()`** una función de perfil previa al onboarding:

```typescript
export type OnboardingPath = 'nuevo' | 'veterano' | 'none';

export function resolveOnboardingPath(): OnboardingPath {
  const configured = localStorage.getItem('gali_goal_configured');
  if (configured) return 'none';                      // ya pasó por onboarding

  const meses = localStorage.getItem('gali_meses_operando');
  const pedidos = Number(localStorage.getItem('gali_pedidos_sem') ?? 0);

  // Veterano: 6+ meses O más de 20 pedidos/sem
  if (meses === '6-12' || meses === '1y+' || pedidos > 20) return 'veterano';

  return 'nuevo';
}
```

**¿Cómo sabe el sistema los meses/pedidos si es la primera vez?**
- Si existe localStorage de una sesión previa de Dropi: usar esos valores
- Si no: el step 2 del onboarding actual los captura — la detección ocurre DESPUÉS de responder el step 2

**Flujo práctico**:
1. Usuario entra por primera vez → no hay `gali_goal_configured` → mostrar step 1 (objetivo) + step 2 (estado actual)
2. Al responder step 2: si `mesesOperando === '6-12' o '1y+'` → saltar a **Camino Veterano** desde el step 3
3. Si `mesesOperando === 'nuevo' y pedidos <= 5` → continuar con **Camino Nuevo**

### 3.2 Camino Nuevo (3 preguntas de alta señal)

**Simplifica el flujo a 3 preguntas** — el resto es contexto que se puede recopilar pasivamente.

**Step 1 — Objetivo** (sin cambio al actual):
- `"¿Qué quieres lograr en los próximos 30 días?"`
- Opciones existentes + texto libre

**Step 2 — Canal principal** (reemplaza el step 2 actual de "estado"):
```
"¿Por cuál canal vendes principalmente?"

○ Meta Ads (Facebook / Instagram)
○ TikTok Ads / TikTok Shop
○ WhatsApp directo
○ No tengo canal activo aún
```

**Step 3 — Mayor fricción** (nuevo, reemplaza steps 3-4 actuales):
```
"¿Cuál es tu mayor problema hoy?"

○ Muchos pedidos con novedades → activa Vigilante
○ No sé qué producto vender   → activa ADA Spy
○ Mi publicidad no da ROAS    → activa Roax
○ No tengo tiempo de atender  → activa Chatea Pro
```

**Resultado inmediato** (lo que hace Gali con las 3 respuestas):

| Objetivo + Canal + Fricción | Módulos visibles | Agente activado |
|---|---|---|
| primer-producto + Meta + qué vender | Productos · Proyectos | ADA Spy |
| escalar-pedidos + Meta + roas | Hub · Marketing · Reportes | Roax |
| primer-producto + WhatsApp + tiempo | Hub · Pedidos · CAS | Chatea Pro |
| automatizar + cualquier + novedades | Hub · Logística · Pedidos | Vigilante |
| cualquier + cualquier + cualquier | Hub completo | Gali |

**Workspace mínimo** — solo 2-3 módulos en el rail después del onboarding:

```typescript
// En GaliWorkspaceService — nueva función
setVisibleModules(modules: string[]): void {
  this.visibleModules.set(modules);
  localStorage.setItem('gali_visible_modules', JSON.stringify(modules));
}

// Ejemplo para "primer-producto":
ws.setVisibleModules(['home', 'productos', 'proyectos', 'cas', 'academy', 'configurar']);
// Ocultos del rail: pedidos, logistica, reportes, financiero, marketing
```

El rail solo muestra los módulos visibles. El usuario puede expandirlos manualmente desde un botón "+ Mostrar más módulos" al fondo del rail.

### 3.3 Camino Veterano (1 confirmación, sin cuestionario)

Si `mesesOperando === '6-12' o '1y+'` después del step 2:

**Reemplazar steps 3-5 por UN solo step de confirmación**:

```
┌───────────────────────────────────────────────────────────┐
│  ✦ Gali                                                   │
│                                                           │
│  Ya sé quién eres.                                        │
│                                                           │
│  Llevas [6 meses] en Dropi con [~47 pedidos/semana].     │
│  Tu objetivo: Escalar de 20 a 50 pedidos.                │
│                                                           │
│  Esto ya lo tengo listo para ti:                         │
│  ✓ Vigilante monitoreando tus pedidos                    │
│  ✓ ROAS real vs. declarado en dashboard                  │
│  ✓ Señales de escala detectadas esta semana              │
│                                                           │
│  [Ir al Hub →]            [Ajustar perfil]               │
└───────────────────────────────────────────────────────────┘
```

- "Llevas X meses" viene de `mesesOperando` del step 2
- "~Y pedidos/semana" viene del slider del step 2
- "Tu objetivo" viene del step 1
- Los 3 items de "Esto ya lo tengo listo" son estáticos — se adaptan al objetivo del step 1

**Comportamiento al pulsar "Ir al Hub →"**:
- `complexityLevel` se pone en `'expert'`
- `visibleModules` se pone como todos los módulos (sin restricción)
- Se cierra el modal y se navega al Hub con `complexityLevel === 'expert'`

**Comportamiento al pulsar "Ajustar perfil"**:
- Retrocede al step 1 para que el usuario pueda cambiar objetivo y datos

## 4. Toggle Vista Simple / Vista Experto en el Hub

### 4.1 Dónde va el toggle

**No** al fondo del scroll. El toggle va en la **Zona 1 del Hub**, en la esquina superior derecha del header de esa zona, junto al título "Decisiones pendientes".

```
┌────────────────────────────────────────────────────────────┐
│  🔴 Decisiones pendientes (2)    [Vista simple | Experto ✓]│
│  ─────────────────────────────────────────────────────     │
│  ...cards...                                               │
└────────────────────────────────────────────────────────────┘
```

El toggle es un **segmented control de 2 opciones** — no un botón que navega, sino un switch inline.

### 4.2 Implementación del toggle

**`home.component.html`** — agregar en el header de Zona 1:

```html

```

### 4.3 Diferencia de contenido según modo

**Vista simple** (`complexityLevel === 'novice'`):
- Zona 1: Decisiones pendientes (máx 2 cards)
- Zona 2: Solo las tarjetas del ciclo con los datos principales (sin sub-métricas)
- Zona 3: Colapsada y no expandible
- Sin "Ciclo de negocio" completo — solo un resumen de 3 líneas
- Sin bloque "Gali esta semana"

**Vista experto** (`complexityLevel === 'expert'`):
- Todo lo definido en Spec #3 (3 zonas completas)
- Ciclo de negocio completo con flechas
- Bloque "Gali esta semana" visible
- Zona 3 expandible

**Implementación en el template** (home.component.html):

```html
<!-- Solo visible en Vista Experto -->
<div class="zone2-full" *ngIf="!isSimpleView()">
  <!-- Ciclo de negocio completo -->
</div>

<!-- Vista simple: resumen compacto -->
<div class="zone2-simple" *ngIf="isSimpleView()">
  <p class="simple-summary">
    {{ proyecto?.pedidos_sem }} pedidos/sem ·
    ROAS {{ proyecto?.roas_real_label }} ·
    Margen {{ proyecto?.margen_neto_pct_label }}
  </p>
</div>
```

// Función pura — mapea las 3 respuestas del onboarding a módulos visibles
function resolveVisibleModules(objetivo: string, friccion: string): string[] {
  const siempre = ['home', 'configurar', 'academy'];

  if (objetivo === 'primer-producto' && friccion === 'que-vender')
    return [...siempre, 'productos', 'proyectos', 'cas'];

  if (objetivo === 'escalar-pedidos' && friccion === 'roas')
    return [...siempre, 'proyectos', 'marketing', 'reportes'];

  if (friccion === 'tiempo')
    return [...siempre, 'pedidos', 'cas'];

  if (friccion === 'novedades')
    return [...siempre, 'proyectos', 'logistica', 'pedidos'];

  // Fallback: rail completo
  return ['home', 'productos', 'proyectos', 'pedidos', 'logistica',
          'marketing', 'reportes', 'financiero', 'cas', 'academy', 'configurar'];
}

finishNuevo(): void {
  const objetivo  = this.selectedGoal()?.id ?? '';
  const friccion  = this.selectedFriccion();             // nueva signal del step 3
  const modules   = resolveVisibleModules(objetivo, friccion);
  this.ws.setVisibleModules(modules);
  this.finish(); // llama finish() existente — persiste gali_goal_configured
}

import PROJECTS from '../../../../../mocks/gali-v5/projects.json';

// Proyecto activo — fuente primaria de datos reales
readonly proyectoActivo = computed(() => {
  const id = localStorage.getItem('gali_proyecto_activo') ?? 'collar-gps-2026';

get pedidosLabel(): string {
