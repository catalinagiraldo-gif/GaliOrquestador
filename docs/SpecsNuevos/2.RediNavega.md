## 1. Diagnóstico — Problemas a resolver

### 1.1 Doble menú sin jerarquía (Problema #1 + Heurística #4)

El rail actual tiene **15 ítems** en tres grupos. El resultado visible es que el usuario ve dos menús distintos sin separación ni etiqueta que explique cuál es cuál:

- **Menú Dropi clásico**: Productos, Pedidos, Logística, Reportes, Financiero, Marketing
- **Menú Gali**: Gali Hub, Proyectos, Agentes, Skills, Reglas, Marketplace, Conexiones

Ítems que **exponen arquitectura técnica interna** y deben salir del rail de primer nivel:

| Key actual | Problema |
|---|---|
| `agentes` | Lenguaje técnico-interno, no de negocio |
| `skills` | El usuario lo percibe igual que "automatizaciones" |
| `reglas` | Se solapa con Skills sin diferencia clara |
| `marketplace` | Conflicto: rail apunta a `/gali-v5/marketplace`, panel usa `/gali-v5/skills/comunidad` |
| `conexiones` | Duplica "Contexto externo" de Mi Negocio |
| `proyectos` | Accesible desde Gali Hub — no necesita ítem propio en el rail |

### 1.2 Tabs del Hub confusas (Problema #4)

- La pestaña "Principal" no comunica que es el dashboard base del negocio del usuario
- Las pestañas de proyecto (ej. "ouiui") no tienen microtexto que identifique qué proyecto es
- El botón "+" no tiene tooltip — el usuario no sabe qué crea

### 1.3 Botones Señales/Lanzar/Medir sin destino (DesADIGali5.md §7.4)

Los botones del workspace mode bar tienen `data-proto-skip` y solo actualizan `ws.activeMode` como signal. No navegan a ninguna ruta. El tooltip `"Prototipo · Ir a modo Señales / Lanzar / Medir"` es visible al usuario final y rompe la inmersión.

| Botón | Efecto real actual | Problema |
|---|---|---|
| ⚡ Operar | Cambia layout a `three-col` | Solo signal, sin URL |
| 🚀 Lanzar | Cambia layout a `two-col` | Solo signal, sin URL |
| 📊 Medir | Cambia layout a `two-col` P&L | Solo signal, sin URL |
| 🔧 Construir | Cambia layout a `two-col` skills | Solo signal, sin URL |
| 🌐 Comunidad | Cambia layout a `full` | Solo signal, sin URL |

Adicionalmente, `home.component.html` L.437: el botón "ver señales" solo llama `gali.requestedPanelTab.set('señales')` — no navega a ninguna vista.

### 1.4 Inconsistencias de routing detectadas (DesADIGali5.md §7.3)

| Problema | Detalle |
|---|---|
| Marketplace duplicado | Rail → `/gali-v5/marketplace` · Paneles de agentes/reglas/skills → `/gali-v5/skills/comunidad` |
| Query param roto | `'/gali-v5/proyectos?nuevo=true'` no tiene guardián — la ruta real es `/gali-v5/proyectos/nuevo` |
| `micromundo` sin rail | Componente existe, no aparece en `DROPI_ICON_RAIL`, solo accesible desde `GALI_MISSION_PANEL` |

## 2. Solución — Modelo Hub-y-Radios

### 2.1 Arquitectura del sidebar

El sidebar pasa de columna plana de 15 ítems a **dos capas**:

```
MACRO-NAV (rail 56px)      MICRO-NAV (panel 200px)
Siempre visible            Contextual al módulo activo
─────────────────────      ──────────────────────────
🔍 Productos               [panel del módulo activo]
🛒 Pedidos
📦 Logística
📊 Reportes
💰 Financiero
📢 Marketing
──────────────
🏠 Gali Hub  ← único ítem IA en el rail
──────────────
🎧 CAS
📖 Academy
⚙️  Configurar
```

Los 6 ítems de orquestación (Agentes, Skills, Reglas, Marketplace, Conexiones, Proyectos) salen del rail y pasan a vivir dentro del panel contextual de Gali Hub como **"Centro de Gali"** — un acordeón colapsable.

### 2.2 Nuevo `DROPI_ICON_RAIL` (10 ítems)

| Key | Label | Grupo | Estado |
|---|---|---|---|
| `productos` | Productos | `main` | Sin cambio |
| `pedidos` | Pedidos | `main` | Sin cambio |
| `logistica` | Logística | `main` | Sin cambio |
| `reportes` | Reportes | `main` | Sin cambio |
| `financiero` | Financiero | `main` | Sin cambio |
| `marketing` | Marketing | `main` | Sin cambio |
| `home` | Gali Hub | `ai` | Único ítem del grupo `ai` — ampliar `matchPrefixes` |
| `cas` | CAS | `utility` | Sin cambio |
| `academy` | Academy | `utility` | Sin cambio |
| `configurar` | Configurar | `utility` | Sin cambio |

**Eliminar del rail**: `proyectos`, `agentes`, `skills`, `reglas`, `marketplace`, `conexiones`

El ítem `home` absorbe todos sus prefijos:

```typescript
{
  key: 'home',
  label: 'Gali Hub',
  icon: 'assets/icons/sidebar/home.svg',
  piIcon: 'pi-home',
  route: '/gali-v5',
  group: 'ai',
  matchPrefixes: [
    '/gali-v5',
    '/gali-v5/proyectos',
    '/gali-v5/proyecto',
    '/gali-v5/agentes',
    '/gali-v5/skills',
    '/gali-v5/reglas',
    '/gali-v5/marketplace',
    '/gali-v5/conexiones',
    '/gali-v5/micromundo',
    '/gali-v5/senales',
  ],
}
```

Con esto `resolveActiveRailKey` retorna `'home'` para todas las rutas de orquestación sin necesidad de la constante `GALI_HUB_PREFIXES` (puede deprecarse o eliminarse).

### 2.3 "Centro de Gali" — acordeón en el panel

El tipo `SectionNavItem` necesita un nuevo valor: `type: 'accordion'`. El componente de panel renderiza ítems accordion como un bloque colapsable con chevron y transición CSS.

**Nuevo `GALI_MISSION_PANEL`**:

```typescript
export const GALI_MISSION_PANEL: SectionPanel = {
  railKey: 'home',
  title: 'Gali',
  items: [
    {
      id: 'gali-hub',
      label: 'Gali Hub',
      route: '/gali-v5',
      icon: 'assets/icons/sidebar/home.svg',
    },
    {
      id: 'nav-proyectos',
      label: 'Proyectos',
      route: '/gali-v5/proyectos',
      icon: `${G5}/boxes.svg`,
    },
    {
      id: 'nav-micromundo',
      label: 'Mi Negocio',
      route: '/gali-v5/micromundo',
      icon: `${G5}/user-circle.svg`,
      badge: 'nuevo',
    },
    {
      id: 'centro-gali',
      label: 'Centro de Gali',
      type: 'accordion',
      icon: `${G5}/settings-sliders.svg`,  // ícono de configuración/sliders
      defaultExpanded: false,
      children: [
        { label: 'Agentes',     route: '/gali-v5/agentes' },
        { label: 'Skills',      route: '/gali-v5/skills' },
        { label: 'Reglas',      route: '/gali-v5/reglas' },
        { label: 'Marketplace', route: '/gali-v5/marketplace' },
        { label: 'Conexiones',  route: '/gali-v5/conexiones' },
      ],
    },
  ],
  agentFooter: {
    agentId: 'gali',
    label: 'Gali',
    color: '#ff6102',
    statusLabel: 'orquestador activo',
    contextKey: 'home',
  },
};
```

**Comportamiento del acordeón**:
- Default colapsado (`defaultExpanded: false`)
- Al expandir: sub-ítems aparecen con transición `max-height` 0 → contenido, 200ms ease-out
- Si una ruta hija está activa (`/gali-v5/agentes`, etc.): el acordeón se fuerza expandido automáticamente
- Chevron del padre rota: `rotate(0deg)` colapsado → `rotate(90deg)` expandido

**Reglas DS aplicadas** (`dropi-sidebar.json`):
- Fondo blanco — `background: #ffffff`
- Ítem activo: sin fondo — solo texto en `#f49a3d` (primary orange)
- Font ítems: Inter Medium 14px, `var(--gray-500, #69738c)`
- Padding: 12px por ítem
- Gap entre ítems: `$size-1` (4px)
- Sub-ítems del acordeón: indentación de `32px` (icono padre 20px + gap 12px = alineados con el texto del padre)
- Íconos: `<img>` tags 20×20px, NO `<i>` PrimeIcons

### 2.4 Transición visual macro-nav → micro-nav

Cuando el usuario cambia de módulo en el rail, el panel contextual debe deslizarse sutilmente para reforzar que está relacionado con el ítem activo:

```scss
.section-panel {
  animation: panel-slide-in 150ms ease-out;
}

@keyframes panel-slide-in {
  from { transform: translateX(-6px); opacity: 0.6; }
  to   { transform: translateX(0);    opacity: 1; }
}
```

El ítem activo del rail tiene indicador lateral:
```scss
.rail-item--active {
  border-left: 2px solid var(--primary-500, #f49a3d);
  // Sin background — DS rule: active item no colored background
}
```

**Archivo**: `home.component.html`

### 3.2 Microtexto bajo pestañas de proyecto

Cada tab que representa un proyecto/dashboard (no "Mi negocio") muestra un microtexto con el nombre del producto principal:

```scss
.tab-subtitle {
  display: block;
  font-size: 10px;
  font-family: Inter, sans-serif;

  line-height: 1.2;
  max-width: 80px;
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
}
```

## 4. Botones Señales/Lanzar/Medir

> **Conflicto resuelto con Spec 4**: Spec 2 **no crea** `SenalesPageComponent` ni placeholder "próximamente". Solo cablea navegación. **Spec 4 es dueño** de la ruta `/gali-v5/senales` y su UI completa.

### 4.1 Señales / Operar → navegación real (contenido en Spec 4)

En lugar de tooltip de prototipo o `gali.requestedPanelTab.set('señales')`:

1. Botón **Operar** (o **Señales** si existe en mode bar) → `router.navigate(['/gali-v5/senales'])`
2. Botón **"Ver señales →"** en Hub → misma ruta
3. Eliminar tooltips visibles con texto `"Prototipo ·"`
4. Quitar `data-proto-skip` de estos botones de navegación

**Hasta que Spec 4 esté implementado**: registrar ruta `/senales` con redirect temporal a Hub **o** dejar ruta lazy que Spec 4 reemplazará — **no** crear componente placeholder intermedio.

Añadir `/gali-v5/senales` a `matchPrefixes` del ítem `home` (§2.2) para que el rail ilumine Gali Hub.

### 4.2 Lanzar y Medir — limpiar `data-proto-skip`

Los botones Lanzar y Medir mantienen su comportamiento de cambiar el layout del Hub (`ws.activeMode`), pero:

1. Se elimina cualquier tooltip o elemento que muestre el texto `"Prototipo ·"`
2. Se elimina el `data-proto-skip` como atributo visible/activo en estos botones (puede dejarse como comentario interno si el equipo lo necesita para tracking)
3. El cambio de modo sigue funcionando — no es un cambio de comportamiento sino de presentación

### 4.3 Botón "ver señales" en home

**Archivo**: `home.component.html` L.437  
Cambiar el handler de `gali.requestedPanelTab.set('señales')` a `router.navigate(['/gali-v5/senales'])`.  
Retirar `data-proto-skip` del botón.
